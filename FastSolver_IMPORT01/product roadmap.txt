# 1. What you are building

You are building a **local Excel add-in product**.

```text
FastSolver.xlam
    = Excel add-in, ribbon button, user interface

FastSolver engine
    = your optimization/automation logic

license.dat
    = local proof that the user has a valid trial or paid license
```

The user opens Excel, clicks your **Fast Solver** button, and your tool pulls data from the workbook, runs the solver/automation, then writes results back into Excel.

# 2. Main goal

The product should work **offline** after activation.

So the normal flow should be:

```text
User pays or starts trial
        ↓
You issue a signed license
        ↓
License is stored locally
        ↓
Excel verifies the license locally
        ↓
Fast Solver works offline until expiration
```

The software should **not** check the internet every time it runs.

# 3. Important correction

You do **not** check “payment” every time.

You check the **local license file** every time.

```text
Every use:
    check local license.dat

Once per year or month:
    user renews payment and gets a new license
```

So “check once a year” really means:

> Check payment/renewal once per year, but check the local license every run.

The local check is fast and offline.

# 4. License file idea

Example `license.dat`:

```json
{
  "license_data": {
    "product": "Fast Solver",
    "email": "user@example.com",
    "license_id": "FS-000123",
    "license_type": "paid",
    "issued_at": "2026-05-16",
    "expires_at": "2027-05-16"
  },
  "signature": "SIGNED_BY_YOU"
}
```

Stored locally, for example:

```text
C:\Users\<user>\AppData\Roaming\FastSolver\license.dat
```

# 5. How the license is protected

Use a **digital signature** system.

You have two keys:

```text
Private key = creates/signs licenses
Public key  = verifies licenses
```

Your license generator has the **private key**.

Your add-in/DLL has only the **public key**.

Formula:

```text
signature = Sign(private_key, license_data)
```

Then the product verifies:

```text
valid = Verify(public_key, license_data, signature)
```

Final decision:

```text
license_ok = valid AND today <= expires_at
```

# 6. Why users cannot edit the date

If the license says:

```json
"expires_at": "2027-05-16"
```

and the user changes it to:

```json
"expires_at": "2099-05-16"
```

then the signed data changed.

The signature no longer matches.

So the software sees:

```text
signature invalid → block
```

# 7. Recommended crypto

Use:

```text
Algorithm: Ed25519
```

Good library choices:

```text
License generator:
    Python + cryptography package

C# DLL:
    NSec.Cryptography

C++ DLL:
    libsodium
```

Do **not** write the cryptography yourself.

# 8. What goes inside the DLL

The DLL can contain:

```text
public verification key
license verification code
expiration check
solver logic
```

The DLL should **not** contain:

```text
private signing key
Stripe secret key
payment API key
database password
customer database
```

If someone reverse engineers the DLL, they may see the **public key**, but that is okay. The public key only verifies licenses; it cannot create them.

# 9. Can someone still crack it?

Yes, theoretically.

A skilled person could patch the DLL or add-in to skip the license check.

But for a cheap product like **$2/month**, most users will not bother.

Your real goal is:

```text
easy to buy
easy to activate
works offline
blocks casual copying
not worth cracking
```

Do not overbuild anti-piracy early.

# 10. Free trial

Yes, you can have a free trial.

A trial is just another signed license with a shorter expiration.

Example:

```json

    "product": "Fast Solver",
    "email": "trial@example.com",
    "license_id": "FS-T-000045",
    "license_type": "trial",
    "issued_at": "2026-05-16",
    "expires_at": "2026-05-30"
  },
  "signature": "SIGNED_BY_YOU"
}
```

Recommended trial:

```text

```

Behavior:

```text
No license → offer trial or activation
Trial valid → run
Trial expired → ask to buy
Paid valid → run
Paid expired → ask to renew
```

Right now you are:

```text

```

That architecture is fine.

# 18. `.exe` issue

If you package Python with PyInstaller, you may get:

```text
FastSolverEngine.exe
```

This can work, but some businesses block unknown `.exe` files.

Especially if they are:

```text
unsigned
unknown publisher
downloaded from internet
not installed by IT
running outside Program Files
```

For individuals and small businesses, this may be fine.

For corporate users, it may need IT approval.

# 19. C++ vs Python

C++ is cleaner for deployment, but businesses do not automatically allow it.

They block based on trust, not language.

They may block:

```text
unknown .exe
unknown .dll
unsigned code
macros
scripts
unapproved apps
```

C++ advantages:

```text
no Python runtime
no pip packages
faster
more professional packaging
easier to make DLL/XLL
```

Python advantages:

```text
faster development
easier solver/data work

```

Best practical path:

```text

Package it cleanly.
Later rewrite critical parts in C++/C# if needed.
```

# 21. Code signing

To look more professional and reduce security warnings, code sign your files.

You may need two kinds of signing:

```text
1. Sign DLL/EXE/MSI
   Windows Authenticode signing

2. Sign the VBA project inside the .xlam
   Office/VBA digital signature
```

Use a real code-signing certificate from a trusted provider.

Common providers:

```text
DigiCert
Sectigo
GlobalSign
SSL.com
Certum
```

For DLL/EXE/MSI signing, use Microsoft SignTool:

```bash
signtool sign ^
  /fd SHA256 ^
  /tr http://timestamp.digicert.com ^
  /td SHA256 ^
  /a ^
  FastSolverCore.dll
```

Verify:

```bash
signtool verify /pa /v FastSolverCore.dll
```

For the `.xlam`:

```text
Excel
→ Developer
→ Visual Basic
→ Tools
→ Digital Signature
→ Choose certificate
→ Save
```
